// 02 — THE_PROBLEM

The Problem

Creator Economy Fragility

Content creators live at the mercy of platform algorithms and corporate policies. A single policy change can erase years of audience building overnight. The creator has no ownership of their digital presence.

GameFi Disconnect

Most blockchain games suffer from either empty gameplay (pure speculation) or no real economy (pure game). The industry lacks experiences that are genuinely fun and financially meaningful.

Fan Engagement Ceiling

Traditional fandom is passive — likes, follows, subscriptions. Fans have no mechanism to actively participate in, protect, or profit from the creators they support most.

NFT Utility Vacuum

The majority of NFT projects deliver digital art with no ongoing utility. Token holders receive no compounding value, no gameplay, and no reason to remain engaged post-mint.

Architecture Principles

  • Server-Authoritative: All game rewards (XP, PTS, items, rank) are computed server-side. The client never sends trust-sensitive values.
  • Idempotency: NFT mints and shop purchases use idempotency keys — duplicate requests are safe and never result in double charges.
  • On-Chain Verification: SOL and USDC payments are verified directly against Solana transaction data before inventory is credited.
  • Zero Stale Closures: All async game state uses React refs to prevent stale closure bugs in realtime callbacks.
  • Error Telemetry: All unhandled errors are captured by Next.js error boundaries and logged to Supabase error_logs via /api/log.
// 06 — NFT_ARCHITECTURE

NFT Architecture

Metaplex Core Standard

All Protect The Streamers NFTs are minted using the Metaplex Core standard via UMI — the next-generation NFT standard on Solana with reduced rent costs and composable plugins. Each card is a unique on-chain asset with fully stored metadata.

Mint Flow

  1. Player connects Phantom wallet
  2. Server generates unique asset keypair + idempotency key
  3. Serialized transaction returned to client
  4. Player signs via Wallet Adapter (0.01 SOL to treasury)
  5. Server confirms on-chain, stores in mint_attempts
  6. NFT secured in player's collection

Mint Safeguards

  • Idempotency key prevents duplicate mints on retry
  • Server checks wallet + streamer uniqueness
  • PENDING_CONFIRMATION state on timeout
  • sessionStorage recovery for interrupted flows
  • Blockhash expiry handled with outer retry loop
  • Treasury verified as fee recipient on-chain

// 09 — PVP_WAGER_SYSTEM

PvP & Wager System

The PvP system is a real-time, server-authoritative turn-based battle engine powered by Supabase Realtime presence channels. Players are matched by wager tier and compete for $PTS, GLR rating points, and leaderboard supremacy.

Matchmaking Flow

  1. Player selects a streamer card and chooses a wager tier (0 / 100 / 500 / 1,000 $PTS)
  2. Player joins the lobby:global Supabase presence channel
  3. Matchmaking scans for opponents with identical wager amounts
  4. Host (lowest session ID) calls initialize_pvp_match RPC — wagers deducted atomically
  5. Match proposal broadcast to matched opponent
  6. If no match within 15 seconds → automatic AI_SENTINEL_V3 bot fallback

GLR Rating System

What is GLR?

Global Leaderboard Rating — an Elo-inspired ranking system that updates server-side after every PvP match via the validate_pvp_move RPC. All GLR calculations happen in PostgreSQL — no client manipulation possible.

GLR Tiers

  • 🟤 Bronze: 0 – 999 GLR
  • ⚪ Silver: 1,000 – 1,999 GLR
  • 🟡 Gold: 2,000 – 3,499 GLR
  • 💎 Diamond: 3,500 – 4,999 GLR
  • 👑 Grand Master: 5,000+ GLR

Wager Security

  • Wagers are deducted atomically during match initialization via server RPC
  • If match initialization fails, automatic refund with 3 retries + exponential backoff
  • Only exact wager-tier matches are made — no cross-tier wager pooling
  • Bot fallback (0 PTS stake) ensures no player is ever stuck in matchmaking
  • PvP forfeit endpoint uses .eq('status', 'ACTIVE') guard — prevents double-forfeit exploits

Turn Timer

A 60-second server-side turn timer (check_turn_timeout RPC) automatically forfeits inactive players. This is enforced at the database level — the client cannot extend or skip the timer.

// 12 — SECURITY_ARCHITECTURE

Security Architecture

Server Authority

No game-critical value (XP, PTS, items, rank, GLR) is ever computed client-side. Every action is verified and computed by a server endpoint or Supabase RPC.

On-Chain Verification

SOL and USDC payments are verified by parsing actual Solana transaction data. The server checks that the treasury received the exact expected amount before crediting inventory.

CSRF Protection

All mutating API routes validate the Origin header against the production domain. Localhost bypass is gated to NODE_ENV !== 'production'.

Optimistic Locking

Shop purchases and mission rewards use updated_at timestamp matching to prevent double-spend race conditions at the database level.

Rate Limiting

All API endpoints are rate-limited: 2s between syncs, 20 logs/min, 60 price requests/min. Prevents both spam and DDoS amplification.

Idempotency

Every mint attempt and shop purchase uses a client-generated idempotency key. The server rejects duplicate keys — retry-safe by design.

Auth Model

  • Wallet signature proves public key ownership (no password required)
  • JWT session stored in HTTP-only cookie (inaccessible to JavaScript)
  • Session cookie cleared on wallet switch or disconnect
  • Login endpoint returns unified error message — no information leakage
  • Session API returns Cache-Control: no-store — prevents stale auth caching
